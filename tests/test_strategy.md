# OBSA 快照测试策略 (Test Strategy)

本文档旨在定义针对 OBSA 插件（华为云对象存储 Hadoop 适配器）快照功能的测试方案。该策略是生成后续测试用例脚本（如 `tests/test_snapshot_lifecycle.py`）的核心指导。

## 1. 测试对象与范围
**目标文件**：`tests/test_snapshot_lifecycle.py` 及其配套的底层引擎 (`dual_runner.py`, `data_mutator.py`)。
**核心功能**：HDFS 快照语意在 OBS 侧的等效实现，包括但不限于开启/禁用快照、创建/删除、差异比对及只读性保护。

---

## 2. 功能测试策略 (Functionality)
验证 OBSA 插件在各种标准操作下是否与原生 HDFS 行为完全一致（双端比对）。

### 2.1 生命周期完整性
- **开启/禁用**：验证 `allowSnapshot` 和 `disallowSnapshot` 在空目录与非空目录下的行为。
- **快照管理**：验证 `createSnapshot`, `renameSnapshot`, `deleteSnapshot` 的成功路径。
- **差异比对**：通过 `snapshotDiff` 验证文件/目录的 `M` (修改), `+` (增加), `-` (删除) 标记是否完全一致。

### 2.2 目录树支持
- **深度递归**：在多层嵌套路径（如 `/a/b/c/d`）上创建快照。
- **混合操作**：在同一个父目录下同时对子目录进行不同的数据变异（删除一些，修改另一些）。

### 2.3 只读隔离性（核心安全点）
- **禁止写入**：严苛测试 `.snapshot/snap_name` 路径下的所有文件/子目录。任何 `touchz`, `put`, `append`, `rm`, `rename` 操作必须被强行拦截并返回一致的 `Permission Denied` 或 `SnapshotException`。

---

## 3. 性能测试策略 (Performance)
关注在大规模数据量和高并发场景下的稳定性与耗时。

### 3.1 吞吐量与并发
- **并发造数**：使用 `DataMutator` 开启多线程（如 50+ 线程）并发向待快照目录注入海量小文件，观察 `createSnapshot` 过程中是否因 metadata 更新而在 OBS 侧产生锁等待或超时。
- **大文件支持**：验证 GB 级别超大文件在快照后的元数据更新效率。

### 3.2 响应耗时 (Latency)
- **快照创建耗时**：随着目录内文件数量从 1k 增加到 100k，对比原生 HDFS 与 OBSA 在 `createSnapshot` 上的耗时曲线。
- **差异遍历耗时**：验证在经过海量变异后，执行 `snapshotDiff` 的检索速度。

---

## 4. 可靠性与鲁棒性策略 (Reliability)
模拟异常和边界情况，确保系统不会处于中间状态。

### 4.1 异常注入
- **网络波动**：模拟在 `createSnapshot` 过程中 OBS 连接中断，验证系统是否能回退或保持元数据一致，不会产生“孤儿快照”。
- **不合理操作**：例如在子目录已开启快照的情况下，尝试通过父目录删除整个路径，验证拦截逻辑。

### 4.2 极限测试
- **快照上限**：验证达到 HDFS 默认单目录 65536 个快照上限时的错误提示一致性。
- **空/超长路径**：测试极长路径名或包含特殊字符的路径下的快照表现。

### 4.3 幂等性与清理
- **重复创建**：验证创建同名快照的错误码。
- **强力清理**：验证 `setup/teardown` 机制在测试崩溃后，能否通过 `rm -rf` 彻底抹除双端环境痕迹，防止测试污染。
